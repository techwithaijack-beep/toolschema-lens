# The hidden tax in agent systems: bloated tool schemas

Most AI teams spend a lot of time debating prompts, models, retrieval quality, and orchestration frameworks.

That makes sense. Those layers are visible.

But there is another layer that quietly adds cost, latency, and instability in production agent systems: the tool contract itself.

Every time an agent request includes OpenAI-style tools, MCP tool definitions, or JSON Schema-based function contracts, the model has to read those structures again. If those contracts are verbose, deeply nested, overloaded with metadata, or stuffed with large enums, the prompt budget starts leaking in a way that is easy to miss.

This is especially common in fast-moving teams. A schema gets written for clarity, then another engineer adds examples, then someone adds titles and defaults, then a product edge case expands the description, then a new nested object arrives for flexibility. Eventually the contract is no longer just a runtime interface. It becomes a mini manual that gets paid for on every turn.

That is the gap behind ToolSchema Lens.

ToolSchema Lens is a local-first CLI that analyzes LLM tool schemas and MCP-style contracts for practical runtime problems:

- tool descriptions that are too long to repeat on every call
- nested structures that make tool use harder for the model to infer
- large enums that consume prompt budget without much reasoning value
- duplicated field descriptions
- metadata that helps human readers more than runtime inference
- underspecified fields that still leave the model guessing

The goal is not to be a perfect optimizer. The goal is to make tool-schema hygiene visible and actionable.

The strongest part of the workflow is how quickly it becomes demoable. You run one command on a JSON file, generate a report, and immediately see before/after token estimates, prioritized findings, and a compacted contract proposal. That makes it useful in architecture reviews, agent performance investigations, pull requests, and MCP cleanup work.

I think this category matters more now because the tooling ecosystem is shifting toward larger multi-tool agent stacks. More MCP servers. More internal tools. More cross-team contracts. More schema surface area. As that happens, contract quality becomes a first-order engineering issue rather than a side detail.

In other words: tool design is becoming prompt design.

The practical lesson is simple. If you want better agent behavior, do not just look at what you ask the model. Look at the interfaces you force it to read every time.

ToolSchema Lens is my attempt to make that problem obvious enough that teams fix it.
